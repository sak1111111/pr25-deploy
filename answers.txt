1. Какие основные различия между development и production окружениями?
Development:

Отладочные инструменты включены

Подробные сообщения об ошибках

Локальные базы данных

Медленная сборка с source maps

Тестовые данные

Production:

Минимизированный и оптимизированный код

Общие сообщения об ошибках (без деталей)

Production базы данных

Быстрая сборка без отладочной информации

Реальные данные

Включено кэширование и сжатие

2. Почему важно использовать environment variables в production?
Безопасность: конфиденциальные данные (API keys, пароли) не попадают в код

Гибкость: разные настройки для разных сред без изменения кода

Масштабируемость: легко менять конфигурацию для разных инстансов

Совместимость: один код работает в разных окружениях

3. Как организовать управление разными средами (dev/staging/prod)?
Разные файлы .env: .env.development, .env.staging, .env.production

Префиксы переменных: DEV_, STAGING_, PROD_

Отдельные ветки в Git: dev, staging, main

Разные серверы/инстансы для каждой среды

CI/CD пайплайны с разными настройками деплоя

4. Какие методы оптимизации производительности при деплое вы знаете?
Frontend:

Минификация и сжатие кода (Webpack, Vite)

Code splitting и lazy loading

Кэширование статических ресурсов

CDN для медиа-файлов

Оптимизация изображений

Backend:

Кэширование запросов (Redis)

Сжатие ответов (gzip)

Балансировка нагрузки

Оптимизация запросов к БД

Пулинг соединений

5. Как настроить мониторинг ошибок в production?
Sentry / LogRocket - для отслеживания ошибок фронтенда

New Relic / Datadog - мониторинг производительности бэкенда

Логи приложения + ELK Stack

Метрики здоровья API (/health endpoint)

Uptime monitoring (UptimeRobot, Pingdom)

Custom alerts при критических ошибках

6. Что такое CI/CD и какие преимущества это дает?
CI (Continuous Integration) - автоматическая сборка и тестирование при каждом коммите

CD (Continuous Deployment) - автоматический деплой после успешных тестов

Преимущества:

Быстрое выявление ошибок

Автоматизация рутинных задач

Стабильность и предсказуемость деплоев

Возможность частых релизов

Снижение человеческого фактора

7. Как организовать процесс обновления приложения без downtime?
Blue-Green Deployment: два идентичных окружения, переключение между ними

Canary Releases: постепенный rollout новой версии

Rolling Updates: поочередное обновление инстансов

Feature Flags: включение функционал без деплоя кода

Миграции БД: backward compatible изменения

8. Какие меры безопасности важно соблюдать при деплое?
HTTPS для всех соединений

Environment variables для секретов

Regular updates зависимостей

Security headers (CSP, HSTS)

Rate limiting и DDoS защита

Database encryption и бэкапы

Access control и принцип наименьших привилегий

Security scanning (dependabot, snyk)
